# Масштабирование баз данных

## Общая стратегия

Сначала применяются оптимизация запросов, вертикальное масштабирование, партиционирование и репликация. Шардирование вводится только после достижения предела одного кластера, поскольку оно значительно усложняет эксплуатацию.

## Репликация
| Сервис | Стратегия |
| --- | --- |
| Auction Engine Service | PostgreSQL primary + standby; Redis Cluster с репликами |
| Campaign Management Service | Primary + асинхронные read replicas |
| Statistics Service | Реплицированные шарды ClickHouse |
| Analytics Service | Реплики ClickHouse для параллельного чтения |
| Financial Service | Primary + synchronous standby; отдельная read replica для истории |

### Ограничения read replicas
Критичные операции не должны читать потенциально устаревшие данные:
- проверка и списание бюджета выполняются на основном экземпляре БД
- чтение сразу после изменения кампании выполняется на основном экземпляре БД
- Auction Engine использует не PostgreSQL-реплики, а версионированные снэпшоты (слепки)

Задержка репликации должна контролироваться, а при превышении порога чтение переключается на основной экземпляр БД.

## Шардирование
| Сервис | Нужно ли|
| --- | --- |
| Auction Engine | Нет на первом этапе |
| Campaign Management | При значительном росте |
| Statistics | Да |
| Analytics | При росте объёма |
| Financial | Только после исчерпания других способов|

Дополнительно Statistics и Analytics партиционируются по времени для ускорения запросов. Все финансовые операции одного пользователя должны находиться в одном шарде. Межшардовые финансовые операции потребуют Saga.

## Read replicas для Analytics
Analytics Service не должен читать реплики Campaign или Financial напрямую. Он строит собственные read-модели из Kafka.

Реплики нужны внутри ClickHouse-кластера Analytics для:
- параллельного выполнения отчётов
- высокой доступности
- разгрузки отдельных узлов
- выполнения длительных запросов без влияния на процессы

## CQRS
CQRS применяется в следующих местах:
- Campaign Management, write-модель кампаний в PostgreSQL, read-модель для RTB в Redis
- Financial, финансовая write-модель в PostgreSQL, оперативные лимиты в Auction Engine Cache
- Statistics, поток сырых событий и отдельные агрегаты
- Analytics, собственные денормализованные read-модели

Команда-владелец сервиса отвечает за его write-модель и события. Команда Analytics отвечает только за витрины и запросы, не изменяя данные доменных сервисов.
