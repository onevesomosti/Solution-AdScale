# Кэширование

## Что кешировать

Данные кампаний:
- активность кампании
- таргетинги
- реклама
- подготовленные кандидаты для аукциона

Ставки:
- активная ставка
- приоритет
- коэффициенты
- версия правил
- минимальная цена

Бюджеты:
- доступный оперативный лимит
- версия состояния
- timestamp обновления
- идентификатор аккаунта

Финансовый баланс в Redis не является источником истины.

## Размещение Redis

Используется отдельный **Redis Cluster**:
- несколько шардов
- master и replica для каждого шарда
- развёртывание отдельно от Auction Engine
- доступ только из Ad Server, Auction Engine и сервисов обновления read-моделей.

Для минимального latency Auction Engine дополнительно хранит слепки данных

## TTL
| Данные | TTL |
| --- | --- |
| Бюджетные лимиты | 30–60 секунд |
| Активные ставки | 1–5 минут |
| Статусы кампаний | 1–5 минут |
| Правила аукциона | до публикации новой версии |
| Справочники  | 1–24 часа |

Чем критичнее актуальность данных, тем короче TTL.

## Инвалидация

Основная стратегия — инвалидация по событию

При получении события:
1. проверяется версия
2. обновляется Redis
3. формируется новый слепок
4. атомарно заменяется старый слепок

TTL используется как дополнительная защита от пропущенных событий.

## Прогрев кэша

Прогрев выполняется:
- при старте нового экземпляра
- после восстановления Redis
- перед ожидаемым пиком
- после публикации крупного набора кампаний

Несколько экземпляров не должны одновременно перегружать БД. 
При недоступности Redis Auction Engine работает на последнем допустимом локальном слепке. Если данные устарели, кандидат исключается или возвращается ответ без контента.
