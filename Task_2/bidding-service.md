# Спецификация Auction Engine Service

## 1. Назначение

**Auction Engine Service** — выделяемый из монолита критичный RTB-сервис, отвечающий за проведение рекламного аукциона и определение победителя.

Сервис выделяется первым среди доменных компонентов, поскольку текущий Auction Engine:
- находится на критичном пути RTB
- наиболее чувствителен к задержкам
- синхронно обращается к общей PostgreSQL
- не может масштабироваться независимо от статистики, аналитики и сервиса рекламы Ad Server Service.

Цель выделения — обеспечить независимое горизонтальное масштабирование Auction Engine, исключить обращения к общей PostgreSQL из критического пути и реализовать драйверы:
- не менее 18 000 RPS через три месяца
- P95 всего RTB-запроса не более 100 ms
- целевую нагрузку до 50 000 RPS
- SLA DSP 99,9%

## 2. Границы сервиса

Auction Engine Service представляет ограниченный контекст проведения рекламного аукциона.

В ответственность сервиса входит:
* приём нормализованного запроса на проведение аукциона
* фильтрация кандидатов
* применение правил аукциона
* расчёт ставок
* проверк бюджетного лимита
* определение победителя
* возврат результатов аукциона

В ответственность сервиса не входит:
| Функция | Владелец |
| --- | --- |
| Приём OpenRTB-запроса от DSP | DSP Integration Service |
| Аутентификация DSP и rate limiting | API Gateway / DSP Integration Service |
| Идентификация пользователя | Ad Server Service |
| Применение таргета и подбор кандидатов | Ad Server Service |
| CRUD кампаний, ставок и таргетингов | Campaign Management Service |
| Управление балансом и финансовые операции | Financial Service |
| Формирование HTML/JS-креатива | Delivery Service |
| Сбор показов и кликов | Statistics Service |
| Формирование отчётности | Analytics Service |

Такое разделение не разрывает Auction Engine на слишком мелкие синхронно зависимые сервисы и не переносит особенности OpenRTB во внутреннюю модель аукциона.

## 3. API

Для Auction Engine выбрана комбинация API:
| Назначение | API |
| Проведение аукциона | gRPC |
| Внешняя интеграция с DSP | OpenRTB через DSP Integration Service |
| Публикация результатов |	Kafka |
| Health checks и метрики |	HTTP |
| Обновление ставок и бюджетов | Kafka → Redis/in-memory |

Такое решение оптимизирует API под характер каждого взаимодействия:
gRPC — для короткого синхронного RTB-пути;
Kafka — для асинхронных событий и обновления read-моделей;
OpenRTB — только на внешней границе;
HTTP — для эксплуатации, служебные эндпойнты не содержат бизнес-функций и используются инфраструктурой.

#### Почему gRPC
- Минимальная задержка. Auction Engine находится на критическом RTB-пути, где время ответа ограничено. Благодаря низким задержкам и компактной передаче данных такой подход удобен для внутренних взаимодействий между сервисами.
- Строгий контракт. Protobuf обеспечивает типизированный API и упрощает поддержку совместимости между сервисами.

## 4. События
После завершения аукциона сервис асинхронно публикует событие. Публикация события не должна задерживать ответ API проведения аукциона. Поэтому публикация событий отделяется от синхронного ответа API.

Потребители:
- Financial Service — финансовый учёт результата
- Statistics Service — статистика аукционов
- Analytics Service — аналитические витрины

## 5. Зависимости
| Зависимость | Назначение | Взаимодействие | Критический путь |
| --- | ---| --- | --- |
| Ad Server Service | Передаёт контекст и кандидатов | gRPC | Да |
| Auction Engine Cache | Ставки, лимиты | Redis / in-memory | Да |
| Campaign Management Service | Источник компаний | Kafka | Нет |
| Financial Service  | Источник бюджетных лимитов | Kafka | Нет |
| Delivery Service   | Получает идентификатор победителя | gRPC | Нет |
| Event Broker | Публикация результатов | Kafka | Да |

## 6. Модель хранения данных
Auction Engine не является владельцем компаний, рекламы или финансовых балансов. Источниками остаются Campaign Management Service и Financial Service.

Сервис хранит только собственную конфигурацию аукциона и данные, необходимые для принятия решения.

#### In-memory snapshot
Основное хранилище на критическом пути. Содержит неизменяемый слепок:
- активных ставок
- бюджетных лимитов
- правил аукциона

#### Redis
Используется как распределённое оперативное хранилище и источник восстановления локального слепка. 

#### Auction Engine DB — PostgreSQL

Результаты каждого аукциона не записываются синхронно в PostgreSQL. Они передаются через Kafka в статистический и аналитический контуры. При необходимости может использоваться вне критического пути. На диаграме целевого решения отсутствует.

## 8. Стратегия выделения из монолита
Выделение выполняется по Strangler Fig Pattern и не требует остановки платформы. 
Шаги:
1. Выделить текущую логику аукциона в изолированный модуль с единым интерфейсом
2. Запретить прямой доступ модуля к таблицам компаний, финансов и статистики
3. Создать локальную read-модель и Redis-кеш.
4. Запустить новый сервис параллельно с монолитом.
6. Переключить небольшую долю трафика через API Gateway.
7. Увеличивать долю при соблюдении latency.
8. После полного переключения удалить старую реализацию Auction Engine из монолита
