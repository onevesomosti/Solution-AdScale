# Проектирование взаимодействий AdScale

## 1. Выбор протоколов

| Взаимодействие | Протокол / паттерн | Обоснование |
| --- | --- | --- |
| DSP → AdScale | HTTPS + OpenRTB | Стандарт внешней RTB-интеграции; DSP передаёт bid request и ожидает bid response |
| DSP Integration → Ad Server | gRPC | Внутренний высоконагруженный вызов с жёстким latency budget |
| Ad Server → Auction Engine | gRPC | Критический RTB-путь, компактный Protobuf, HTTP/2, deadline propagation |
| Auction Engine → Delivery | gRPC | Результат нужен синхронно для формирования bid response |
| Advertiser Dashboard → Campaign Management | REST/HTTPS | CRUD-операции, удобство браузерных клиентов и документирование через OpenAPI |
| Campaign Management → другие сервисы    | Kafka | Распространение изменений кампаний, ставок и таргетингов |
| Financial → Auction Engine | Kafka | Асинхронное обновление бюджетных лимитов без вызова Billing в RTB-пути |
| Показы, клики, результаты аукционов | Kafka Event Streaming | Большой поток событий, несколько независимых потребителей, возможность повторного чтения |

Критичный RTB-поток должен оставаться коротким и синхронным, а статистика и аналитика не должны задерживать ответ DSP. 

## 2. Обоснование
### Почему gRPC для Bidding
Для взаимодействия с Auction Engine выбирается unary gRPC:

Причины:
- Auction Engine находится на наиболее чувствительном к задержкам участке
- Protobuf компактнее JSON и быстрее сериализуется
- gRPC предпочтителен для частых внутренних вызовов с жёсткими требованиями по задержкам

### Почему REST для Campaign Management Service
Campaign Service обслуживает рекламный кабинет и выполняет преимущественно CRUD-операции.

Для него REST удобнее, потому что:
- API ресурсоориентированный
- его легко вызывать из браузера
- контракт можно документировать в OpenAPI
- требования по latency значительно мягче, чем у Auction Engine
- запросы удобно отлаживать через стандартные HTTP-инструменты

### Почему Kafka для событий
Клики, показы и результаты аукционов являются не командами, а фактами. Kafka подходит для них потому что:
- поддерживает высокую пропускную способность
- сохраняет события в течение заданного срока
- позволяет повторно прочитать поток
- несколько consumer groups независимо получают одно событие
- партиции позволяют сохранить порядок событий
- брокер буферизует пики и отделяет RTB от статистики и аналитики

## 3. API Gateway
API Gateway нужен как единая точка входа для внешнего трафика. Его функции:
- TLS теминации
- маршрутизация OpenRTB и REST-запросов
- rate limiting по DSP и рекламодателю
- проверка API key/JWT
- защита внутренних сервисов от прямого внешнего доступа

Gateway не должен содержать бизнес-логику или агрегировать данные, иначе он станет узким местом.

## 4. Нужен ли Service Mesh

На первом этапе Service Mesh внедрять не следует:
- сервисов пока немного
- контейнеризация и CI/CD ещё формируются
- нет выделенной SRE-команды
- увеличит операционную сложность

Timeout, retry, circuit breaker и tracing первоначально реализуются средствами клиентов gRPC, Gateway и общих библиотек.

Service Mesh можно внедрить позже, когда количество сервисов и межсервисных связей существенно вырастет.
