# API-слой интеграции с DSP

API Gateway выполняет инфраструктурные функции, а DSP Integration Service отвечает за специфику конкретного партнёра и преобразование OpenRTB во внутреннюю модель AdScale.

API Gateway:
- завершает TLS-соединение
- проверяет идентификатор DSP
- выполняет аутентификацию и rate limiting
- добавляет нужные внутренние связи (FK)

DSP Integration Service:
- валидирует OpenRTB-запрос
- применяет настройки конкретной DSP
- преобразует запрос во внутреннюю модель
- формирует OpenRTB bid response

Gateway не должен содержать бизнес-логику аукциона иначе он станет узким местом.

### Ограничение частоты запросов

Rate limit реализуется централизованно на API Gateway

Лимиты задаются отдельно:
- для каждой DSP
- для каждого API-ключа
- для каждого региона
- для эндпойнта

При превышении лимита Gateway возвращает:
```http
HTTP/1.1 429 Too Many Requests
```
Дополнительно на DSP Integration Service устанавливается локальный лимит для защиты от перегрузки Gateway или ошибочной конфигурации глобального лимита.

## Аутентификация

Основной механизм — **mTLS**:
- DSP и AdScale обмениваются клиентскими сертификатами
- Gateway проверяет сертификат партнёра
- сертификат связывается с внутренним dsp_id
- поддерживаются ротация и отзыв сертификатов

Дополнительный механизм — API key. Используется для идентификации интеграции, ротации полномочий и раздельного ограничения трафика.

Проверки выполняются на Gateway:
1. сертификат выдан доверенным центром, не отозван и не просрочен
3. API key активен
4. dsp_id соответствует сертификату и ключу
5. источник запроса входит в разрешённый список сетей, если DSP предоставляет фиксированные адреса

JWT для каждого bid request не используется: для высокочастотного server-to-server RTB-трафика mTLS и API key проще и создают меньше накладных расходов.

Для RTB-трафика между сервисами mTLS и API key проще в реализации и поддержке, поэтому JWT тут не используется.

## Circuit Breaker

Circuit Breaker устанавливается на вызовах:
- API Gateway → DSP Integration Service
- DSP Integration Service → Ad Server Service
- Ad Server Service → Auction Engine Service
- Auction Engine Service → Delivery Service

Состояния:
- Closed, запросы проходят
- Open, вызовы временно блокируются
- Half-Open, пропускается небольшое число определенных запросов

Точные параметры настоек уточняются после с помощью тестирования.

### Fallback

При размыкании цепи RTB-запрос не ставится в очередь и не ждёт восстановления сервиса. Возвращается корректный ответ с подходящим HTTP кодом или OpenRTB-ответ без данных (согласно контракту DSP). Это защищает потоки от накопления запросов и предотвращает каскадный отказ. Circuit Breaker быстро блокирует вызовы к стабильно недоступному сервису и периодически проверяет его восстановление.

### Retry
Автоматические повторы в RTB hot path ограничиваются:
- не более одной повторной попытки
- при ошибке установления соединения до передачи запроса
- без повтора для ошибок валидации и после получения ответа от Auction Engine

Полный повтор запросов может увеличить задержки и создать дубли. Поэтому основная стратегия — быстрый отказ, а не многократный retry.

## 8. Применимость BFF и Saga

### BFF
Отдельный BFF для DSP не нужен. BFF применяется, когда требуется агрегировать и адаптировать данные под особенности пользователя. Для DSP эту задачу выполняет DSP Integration Service.

### Saga
Saga не применяется к обработке одного запроса:
- запрос не является длительной распределённой транзакцией
- результат должен быть получен за десятки миллисекунд
- при ошибке используется ответ без контента

Saga потребуется в отдельных финансовых процессах, например при резервировании, списании и последующей корректировке бюджета между Auction и Financial. Паттерн предназначен для цепочки локальных транзакций с компенсирующими операциями, а не для маршрутизации синхронного RTB-запроса.

## Итоговое решение

| Задача | Решение |
| --- | --- |
| Внешний API | HTTPS + OpenRTB|
| Точка входа | Gateway |
| Адаптация DSP | DSP Integration Service |
| Маршрутизация | По эндпойнтам и dsp_id |
| Rate limiting | На Gateway |
| Аутентификация | mTLS + API key |
| Защита от каскадных сбоев | Circuit Breaker + быстрый no_content |
| Retry | Не более одной безопасной попытки |
| BFF | Не применяется для DSP |
| Saga | Не применяется в RTB hot path |

Такой API-слой минимизирует работу Gateway и сохраняет бизнес-логику внутри сервисов:
- Gateway защищает и маршрутизирует трафик
- DSP Integration Service адаптирует OpenRTB
- Auction Engine проводит аукцион

### Обеспечение надёжности
- **Circuit Breaker.** Для вызовов зависимых сервисов используется автоматический выключатель. При росте ошибок или времени ответа он размыкает цепь и временно прекращает вызовы, предотвращая каскадный отказ.

- **Retry.** Повторные попытки применяются только для временных сетевых ошибок и 5xx ответов. Используется экспоненциальная задержка и ограниченное число попыток.

- **Идемпотентность финансовых операций.** Резервирование и списание бюджета выполняются с уникальным `idempotency_key`, например на основе `auction_id` и типа операции. Повторный запрос не должен создавать повторное списание или резервирование.

- **Резервные стратегии.** При недоступности связанных сервисов аукцион использует последний актуальный слепок данных из Redis или локального кеша. Если снимок отсутствует или устарел, применяется безопасная дефолтная стратегия: исключение кандидата либо возврат ответа без данных.

- **Контроль актуальности.** Для кешированных данных задаются версия, timestamp и допустимый возраст. Использование устаревших финансовых данных должно быть ограничено, чтобы не допустить перерасхода бюджета.
