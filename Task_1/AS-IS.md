# AdScale. Анализ текущей архитектуры

## Описание системы (из задания)
| Категория	| Технология |
| --- | --- | 
|Backend |	C++ / Go / Python / Node (гибридный монолит) |
|База данных |	PostgreSQL (одна инстанция) |
|Кеш |	Отсутствует |
|Очереди |	Отсутствуют |
|Контейнеризация |	Нет |
| Деплой |	Вручную |
| Инфраструктура |	Один production-кластер |

Нагрузочный профиль:
- До 15 000 RPS в пике на аукцион;
- Строгие требования по времени отклика: не более 100 ms для аукциона;
- Высокая нагрузка на запись (события кликов и показов);
- Постоянный поток событий (event stream).

## Ключевые проблемы

| Зона | Проблема |	Риски |
| --- | --- | --- |
| Auction Engine |	Синхронные вызовы к БД в hot path |	Рост задержек , проигрыш аукционов |
| PostgreSQL |	Одна инстанция для ставок, финансов, аналитики. Конкуренция за ресурсы БД. |	Single Point of Failure, проблемы транзакций  |
| Statistics Service |	Прямая запись кликов/показов без буферизации |	Bottleneck |
| Analytics Service |	Тяжёлые SQL-запросы в production-БД, нет read-replica	| Блокировки и рост latency |
| Ad Server / Auction Engine/ Delivery Service | Критичный бизнес-поток тесно связан внутри монолита | Невозможно независимо масштабировать ключевые сервисы, отказ одной системы приводит к отказу всего. |
| Деплой | Нет контейнеризации, деплой вручную | Высокий операционный риск |
| Документация |	Архитектура не задокументирована |	Зависимость от знаний команды |

Текущая архитектура не соответствует будущей нагрузке и SLA DSP-партнёра.
