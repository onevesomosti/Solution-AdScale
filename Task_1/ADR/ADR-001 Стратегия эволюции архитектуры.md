# ADR-001: Стратегия эволюции архитектуры AdScale 

**Дата:** 2026-07-11

## Контекст

Система AdScale представляет собой гибридный монолит с общей базой данных. Платформа работает под нагрузкой до **15 000 RPS**, а после подключения новой DSP ожидается рост до **18 000 RPS** с дальнейшим увеличением до **50 000 RPS**. Полная переработка системы невозможна из-за ограничения в **3 месяца**, ограниченного бюджета и отсутствия полноценной DevOps-инфраструктуры.

**Ключевые архитектурные драйверы:**
- поддержка роста нагрузки и независимого масштабирования;
- снижение влияния аналитики и статистики на RTB-контур;
- обеспечение P95 ≤ 100 ms и SLA 99,9%;
- непрерывность бизнеса при миграции;
- готовность к дальнейшему развитию архитектуры.

Рассматривались три альтернативы:
1. Оставить в составе единого монолита
2. Переход сразу на микросервисы
3. Поэтамная миграция через модульный монолит в микросервисы

### Альтернатива 1: Единый монолит
**Плюсы:** минимальные изменения, быстрый результат.
**Минусы:** отсутствует независимое масштабирование, сохраняется единая точка отказа и конкуренция за ресурсы общей базы данных.

### Альтернатива 2: Переход на микросервисы
**Плюсы:** независимое масштабирование, изоляция компонентов, соответствие целевой архитектуре.
**Минусы:** высокий риск, необходимость построения новой инфраструктуры, невозможность реализации за три месяца.

### Альтернатива 3: Поэтапная миграция через модульный монолит
**Плюсы:** постепенная миграция без остановки работы, снижение рисков, поэтапная подготовка к микросервисной архитектуре.
**Минусы:** временное сосуществование старой и новой архитектуры, увеличение сложности сопровождения.

## Решение

Принята стратегия **поэтапной миграции** с использованием **Strangler Fig Pattern**.

На первом этапе монолит преобразуется в **модульный монолит** с выделенными границами доменных модулей и взаимодействием через публичные интерфейсы. Наиболее нагруженные компоненты (Auction Service, Statistics Service, Analytics Service) постепенно выделяются в самостоятельные сервисы.

При реализации решения применяются следующие принципы:
- чёткие границы модулей и запрет прямого доступа к данным других модулей
- API Gateway для постепенного переключения трафика
- Kafka для асинхронного обмена событиями
- Database per service при выделении микросервисов
- Независимое масштабирование критичных сервисов

## Последствия

**Положительные:**
* миграция без остановки бизнеса
* независимое масштабирование наиболее нагруженных компонентов
* снижение влияния аналитики на RTB-контур
* поэтапная подготовка к целевой микросервисной архитектуре

**Отрицательные:**
* необходимость сопровождать монолит и новые сервисы одновременно
* увеличение сложности интеграции
* постепенный переход к eventual consistency

**Риски:**
* образование распределённого монолита при неправильном определении границ сервисов
* рост сложности эксплуатации по мере увеличения числа сервисов
