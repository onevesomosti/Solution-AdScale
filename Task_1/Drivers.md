# AdScale. Архитектурные драйверы

### Функциональные требования

#### Бизнес-цели:
- подключить DSP-площадку без деградации SLA, 
- поддерживать не менее 18 000 RPS
- P95 не выше 100 ms, 
- обеспечить отказоустойчивость сервиса ставок
- масштабировать нагруженные компоненты
- разделить критичные/некритичные потоки
- поддержать мультиплатформенные интеграции

#### Требования к промежуточному решению (через 3 месяца):
1. Обрабатывать зпросы от новой DSP-площадки по OpenRTB.
2. Обрабатывать RTB-аукционы.
3. Разделить критичный поток RTB и некритичные потоки: статистика, аналитика, кабинет.
4. Буферизовать события показов и кликов перед записью в хранилище.
5. Перенести тяжёлую аналитику с production-БД на read-replica или отдельное хранилище.
6. Обеспечить контроль метрик.

#### Требования к целевому решению (через год)
1. Обеспечить мультиплатформенную интеграцию.
2. Перейти на микросервисную архитектуру и событийно-ориентированную обработку. Обеспечить возможность для независимого масштабирования сервисов.
3. Подготовить систему к мультиплатформенной интеграции.
4. Обеспечить мультирегиональное развёртывание.

### Атрибуты качества
Новый партнёр (DSP-площадка) вводит более жёсткий SLA, штрафы за latency и downtime, а рост трафика ведёт к увеличению нагрузки на движок ставок, запись событий и риск таймаутов.

| Приоритет	| Атрибут |	Целевое состояние |
| --- | --- | --- |
| P1 |	Performance  |	P95 ≤ 100 ms для исходного решения; P95  ≤ 80 ms для DSP-партнёра |
| P1 |	Scalability |	18 000 RPS через 3 месяца; 50 000 RPS через год. |
| P1 |	Availability |	Не допускать простоев, влияющих на DSP-интеграцию; целевой SLA через год — 99,9%. |
| P1 |	Reliability |	Auction Engine не должен сбоить; события кликов/показов не должны теряться, транзакции должны быть атомарны. |
| P1 |	Reliability |	Таймауты DSP, БД или аналитики не должны замедять критичный RTB-поток |
| P2 | 	Security |	Защита финансовых операций, кабинета рекламодателя и интеграционных API. |
| P2 |	Modifiability |	Возможность подключения новых DSP без дополнительной разработки. |
| P2 | Maintainability	 | Снижение риска ручного деплоя, документирование |
| P3 | Observability | Метрики, логи, трассировки |

### Ограничения

#### Технические ограничения
| Драйвер | Значение |
| --- | --- |
| Стек | Гибридный монолит C++ / Go / Python / Node.js |
| Стек | PostgreSQL — одна инстанция | 
| Стек | Отсутствует кэш | 
| Стек | Отсутствуют очереди |
| Стек | Интеграция по протоколу OpenRTB |
| Инфраструктура | Один production-кластер |
| Инфраструктура | Ручной деплой |
| Команда | Нет выделенного SRE |

#### Бизнес-ограничения
| Драйвер | Значение |
| --- | --- |
| Сроки | 3 месяца для интеграции с новой DSP-площадкой|
| Бюджет |	Невозможно резко расширить команду или полностью переписать систему |
| Непрерывность | Нельзя останавливать платформу на время изменений | 
| Performance | Штрафы от DSP |
| Performance | Новые интеграции не должны ухудшить сервис для существующих клиентов |
